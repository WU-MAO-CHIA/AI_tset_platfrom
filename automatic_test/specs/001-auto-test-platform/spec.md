# Feature Specification: 自動化測試平台 (Automatic Test Platform)

**Feature Branch**: `001-auto-test-platform`  
**Created**: 2026-05-14  
**Status**: Draft  
**Last Updated**: 2026-05-22  
**Input**: PRD.md — 自動化測試平台，涵蓋測案管理（含 CRUD、AI 輔助步驟補齊、多媒體上傳、建立後即時試跑、Excel/CSV 測試參數匯入、測試清單建立與執行紀錄）、資料庫串接、AI 自動化代碼生成、平行執行網頁測試、測試結果網頁呈現（含影片/圖片）及測試報告產生

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 建立測試案例 (Priority: P1)

測試人員需要在平台上建立新的測試案例，填寫完整欄位後儲存至測試案例資料庫，供未來不同專案和測試人員使用。

**Why this priority**: 測試案例資料庫是整個平台的核心基礎，所有後續功能（清單管理、測試執行）都依賴可重複使用的測試案例存在。

**Independent Test**: 透過「建立測試案例」畫面填寫所有欄位並儲存，驗證案例出現在清單中且版本號為 v1，即可作為獨立驗收。

**Acceptance Scenarios**:

1. **Given** 測試人員進入「建立測試案例」畫面，**When** 填寫所有必填欄位（編號、名稱、說明、主要步驟、系統別）並點擊儲存，**Then** 系統儲存該案例並自動設定版本號 v1，案例出現在測試案例清單中。
2. **Given** 已存在的測試案例，**When** 測試人員修改任意欄位並儲存，**Then** 系統自動遞增版本號（如 v1 → v2）並記錄修改人員。
3. **Given** 建立測試案例表單，**When** 測試人員點擊「新增測試資料」按鈕，**Then** 測試資料表格末尾新增一空白列，含四個行內輸入欄位：**易讀名稱**（自由文字）、**RF 變數**（自動帶入 `${易讀名稱}`，可手動覆寫）、**預設值**、**說明（備註）**；可重複操作新增多筆，並可行內直接編輯或刪除任意列。
4. **Given** 測試人員提交表單，**When** 必填欄位為空，**Then** 系統顯示對應欄位的錯誤提示，禁止儲存。
5. **Given** 測試人員在建立測試案例時填寫部分步驟或說明，**When** 點擊「AI 補齊」按鈕，**Then** 系統以管理後台設定的全域預設 LLM 模型（見 FR-027）呼叫 AI，根據已填內容自動補充或完善測試步驟，測試人員可接受或修改建議內容。
6. **Given** 測試人員在建立測試案例畫面，**When** 上傳圖片、影片或貼上網址作為輔助資訊，**Then** 系統接受上傳內容並顯示在表單中，LLM 可將該多媒體內容作為理解測試步驟的上下文。
7. **Given** 全域預設 LLM 模型由 `.env` 的 `DEFAULT_LLM_MODEL` 設定，**When** 測試人員觸發 AI 補齊或 RF 代碼生成，**Then** 系統一律以該全域預設模型執行；建立案例表單不提供個別模型切換器。
8. **Given** 測試案例已存在於資料庫，**When** 測試人員點擊刪除並確認，**Then** 系統將該案例從清單中移除，後續無法再被選取或查詢。
9. **Given** 測試案例已被一個或多個測試清單引用，**When** 測試人員嘗試刪除該案例，**Then** 系統顯示受影響的清單名稱並要求二次確認，確認後才執行刪除。
10. **Given** 測試人員完成測試案例所有欄位填寫並儲存，**When** 點擊「立即試跑」，**Then** 系統使用 AI 將測試步驟轉譯並立即執行一次，完成後在畫面上顯示執行結果（通過/失敗、截圖、執行時間），供測試人員確認案例是否符合預期。
11. **Given** 試跑結果與測試人員預期不符，**When** 測試人員關閉試跑結果並修改步驟後再次儲存，**Then** 可重新觸發試跑，歷次試跑紀錄不計入正式測試清單的執行歷史。
12. **Given** 測試人員在建立或編輯測試案例畫面，**When** 頁面完整載入，**Then** 畫面以 Tab 分頁呈現：Tab 1「基本資訊」顯示基本欄位及儲存按鈕；Tab 2「測試步驟」以左右分割顯示，左側為 AI 對話區（Chat 介面），右側為 RF 程式碼預覽區。
13. **Given** 測試人員在 Tab 2 左側 Chat 輸入框送出訊息，**When** AI 回應完成，**Then** AI 回應以對話氣泡顯示於左側對話區，右側 RF 程式碼自動更新為最新生成結果，無需額外點擊。
14. **Given** 測試人員儲存測試案例，**When** Tab 2 有對話歷史，**Then** 對話歷史與案例一併儲存，下次開啟該案例進入 Tab 2 時可繼續對話。
15. **Given** 測試案例已有已儲存的 RF 程式碼，**When** 測試人員進入編輯模式，**Then** 系統自動解析 RF 程式碼中所有非 RF 內建的 `${VARIABLE_NAME}` 參照，並以合併補齊策略將尚未存在於測試資料表格的變數新增（`rf_variable` 設為 `${VARIABLE_NAME}`，`field_name` 設為去除 `${}` 後的識別名稱），已存在的列不覆蓋；若案例尚無已儲存的 RF 程式碼，則略過此步驟。
16. **Given** 測試人員在 Tab 2「測試步驟」右側 RF 程式碼預覽區已有程式碼內容，**When** 點擊「立即試跑」按鈕，**Then** 系統直接使用當前預覽區的 RF 程式碼（不需先儲存案例）執行試跑；完成後將試跑結果（通過/失敗 badge、錯誤訊息、執行時間、最多 1-3 張失敗截圖）以 `type: "trial_run_result"` 訊息附加至左側 Chat，並自動觸發 AI 分析失敗原因、回應建議修正的 RF 程式碼；此結果訊息持久化至案例對話歷史（DB），下次開啟可見。

### User Story 2 - 瀏覽與篩選測試案例 (Priority: P1)

測試人員需要從測試案例資料庫清單中快速找到符合條件的案例，並查看詳細內容。

**Why this priority**: 隨著案例數量增加，有效的搜尋與篩選機制直接影響測試人員的工作效率，與建立案例同等重要。

**Independent Test**: 在測試案例清單頁面使用多條件篩選找到目標案例，並點入查看完整詳情，即可獨立驗收。

**Acceptance Scenarios**:

1. **Given** 測試案例清單頁面包含多個案例，**When** 測試人員選擇特定「系統別」進行篩選，**Then** 清單只顯示符合該系統別的案例。
2. **Given** 測試案例清單，**When** 測試人員同時設定「系統別」與「名稱關鍵字」兩個篩選條件，**Then** 清單只顯示同時符合兩個條件的案例。
3. **Given** 篩選完成的清單，**When** 測試人員點擊某個案例，**Then** 進入詳細畫面，以 Tab 分頁呈現：Tab 1「基本資訊」顯示完整欄位與版本歷史（唯讀），Tab 2「測試步驟」顯示 AI 對話歷史與最新 RF 程式碼（均為唯讀）。
4. **Given** 篩選條件無符合結果，**When** 搜尋完成，**Then** 清單顯示「無符合結果」的提示訊息。
5. **Given** 測試人員在測試案例清單或詳細畫面，**When** 上傳 Excel、CSV 或文字檔作為測試參數，**Then** 系統解析檔案內容並將資料填入測試案例的測試資料欄位，測試人員可預覽並確認後儲存。
6. **Given** 測試案例清單或詳細畫面，**When** 測試人員點擊某個案例的執行歷史入口，**Then** 系統顯示該案例參與過的所有測試清單執行紀錄，並可點入查看各次執行的結果詳情。

### User Story 3 - 測試清單管理 (Priority: P2)

測試人員可以建立測試清單，填寫清單名稱與建立人員，從測試案例清單中挑選所需案例匯入，並可在清單中查閱每次執行的歷史紀錄。

**Why this priority**: 測試清單將資料庫中的案例組合成可執行的測試任務，並透過執行紀錄追蹤歷次測試結果，是平台核心工作流程的橋樑。

**Independent Test**: 建立一個有名稱與建立人的測試清單、加入若干案例並儲存後，清單內容完整呈現，即可獨立驗收。

**Acceptance Scenarios**:

1. **Given** 測試人員在「建立測試清單」畫面，**When** 填寫清單名稱與建立人員，並從測試案例清單中選取案例匯入，**Then** 清單以所填名稱儲存，所選案例顯示於清單中。
2. **Given** 已建立的測試清單，**When** 測試人員移除某個案例，**Then** 該案例從清單中移除，其他案例不受影響。
3. **Given** 已執行過的測試清單，**When** 測試人員查看該清單，**Then** 頁面顯示每次執行的歷史紀錄，包含執行時間與結果摘要。
4. **Given** 測試清單有多次執行紀錄，**When** 測試人員點擊某筆紀錄，**Then** 顯示該次執行的詳細結果。
5. **Given** 測試人員在任意頁面（如「測試清單」），**When** 點擊全域導覽列的「測試案例管理」連結，**Then** 系統導覽至 /cases 頁面，導覽列高亮標示目前所在頁面，其餘頁面狀態不受影響。
6. **Given** 測試人員在清單詳情頁（/checklists/:id），**When** 點擊「管理案例」按鈕，**Then** 系統導覽至案例管理畫面（/checklists/:id/cases），顯示目前清單中所有案例及其排序。
7. **Given** 案例管理畫面，**When** 測試人員點擊「新增案例」，**Then** 系統開啟案例選取介面，可從現有案例清單中搜尋並選取，選取後案例加入清單並顯示在列表末尾；搜尋時**同時比對案例編號（case_number）與案例名稱（name）**，任一欄位符合關鍵字即列入結果。
8. **Given** 案例管理畫面，**When** 測試人員拖曳案例列或點擊上/下移動按鈕，**Then** 清單中案例順序即時更新，儲存後執行時依此順序執行。
9. **Given** 案例管理畫面，**When** 測試人員在某案例列新增備註，**Then** 備註儲存為清單層級屬性（僅此清單可見），不影響原始案例內容；案例本身欄位（步驟、說明等）在此畫面為唯讀，提供「前往編輯案例」連結跳轉至 /cases/:id/edit。
10. **Given** 清單詳情頁，**When** 測試人員點擊「編輯清單」，**Then** 可修改清單名稱與建立人員，儲存後詳情頁即時更新。
11. **Given** 清單詳情頁，**When** 測試人員點擊「刪除清單」並確認，**Then** 系統執行刪除，若有執行中的測試正在使用此清單，系統禁止刪除並顯示提示。

### User Story 4 - 串接資料庫撈取測試資料 (Priority: P3)

測試人員可設定外部資料庫連線，執行查詢撈取真實測試資料，並套用至測試案例的測試資料欄位。

**Why this priority**: 自動化撈取真實資料可大幅降低手動準備測資的工作量，但此功能相依於測試案例資料庫已建立完成。

**Independent Test**: 設定資料庫連線、執行查詢，並將結果成功套用至一個測試案例的測試資料欄位，即可獨立驗收。

**Acceptance Scenarios**:

1. **Given** 測試人員填寫資料庫連線資訊，**When** 點擊「測試連線」，**Then** 系統回傳連線成功或失敗的明確提示。
2. **Given** 已建立連線的資料庫，**When** 測試人員輸入查詢條件並執行，**Then** 查詢結果以可選取的列表呈現。
3. **Given** 查詢結果列表，**When** 測試人員選取資料並套用至測試案例，**Then** 測試案例的測試資料欄位填入對應資料。

### User Story 5 - 執行網頁測試與產生報告 (Priority: P3)

測試人員可選取測試清單執行自動化網頁測試，並於執行完成後產生可查閱與下載的測試報告。

**Why this priority**: 這是平台的最終目標，但需要前述所有功能（案例、清單、測試步驟）均已就緒才能有意義地執行。

**Independent Test**: 選取一個測試清單執行，執行完成後成功產生並下載測試報告，即可獨立驗收。

**Acceptance Scenarios**:

1. **Given** 已建立的測試清單且清單中案例數 > 0，**When** 測試人員點擊「執行測試」，**Then** 系統開始依序執行清單中各案例的測試步驟，並顯示即時進度。
1a. **Given** 測試清單中無任何案例（案例數 = 0），**When** 測試人員點擊「執行測試」，**Then** 系統顯示警示訊息「測試清單為空，請先新增案例」，禁止執行，不觸發後端請求。
2. **Given** 測試執行中，**When** 某個案例執行失敗，**Then** 系統記錄失敗原因與截圖，並繼續執行後續案例。
3. **Given** 所有案例執行完畢，**When** 測試人員開啟報告，**Then** 報告顯示每個案例的執行結果（通過/失敗）、執行時間及失敗詳情。
4. **Given** 測試報告已產生，**When** 測試人員點擊「匯出報告」，**Then** 系統提供可下載的報告檔案。
5. **Given** 測試清單中有多個測試案例，**When** 測試人員啟用平行執行模式並開始執行，**Then** 系統同時執行多個測試案例，執行進度畫面分別顯示每個案例的即時狀態。
6. **Given** 測試案例包含自然語言撰寫的測試步驟，**When** 系統準備執行該案例時，**Then** AI 自動將自然語言步驟翻譯為可執行的自動化測試代碼，執行完成後報告中標注所使用的代碼版本。
7. **Given** 測試執行完成，**When** 測試人員開啟執行結果頁面，**Then** 系統以網頁形式呈現結構化執行結果，包含每個案例的通過/失敗、執行時間及失敗詳情，並於頁面中內嵌顯示執行過程中擷取的截圖或影片。
8. **Given** 執行結果頁面，**When** 測試案例有多筆擷取的截圖或影片，**Then** 所有媒體依測試步驟順序排列並可在頁面上直接瀏覽，無需另行下載。

### Edge Cases

- 測試案例編號重複時，系統應拒絕建立並提示重複錯誤。
- 測試資料欄位在合理使用範圍內支援動態新增，無硬性筆數上限。
- 外部資料庫連線逾時時，系統應顯示逾時提示並允許重試，已撈取資料不受影響。
- 測試執行中若目標網頁無回應，系統應記錄逾時錯誤並繼續後續案例。
- 若清單中案例在執行前已被修改，系統應使用最新版本執行，並在報告中標注版本號。
- AI 補齊呼叫超時或 LLM API 回傳錯誤時，系統應顯示明確提示，已填寫內容不受影響，測試人員可選擇重試或手動填寫。
- 上傳媒體附件超過大小或格式限制時，系統應拒絕上傳並顯示對應提示，不影響其他欄位的已填內容。
- 貼入的網址無效或無法存取時，系統應顯示警告提示，但不阻擋測試案例儲存（網址僅為輔助上下文）。
- 刪除測試案例時，若有進行中的測試清單執行正在使用該案例，系統應禁止刪除並提示當前執行狀態。
- 點選「執行測試」時，若測試清單中案例數量為 0，系統應在前端攔截並顯示警示提示「測試清單為空，請先新增案例」，不觸發後端執行請求。
- 刪除確認對話框中，使用者點擊「確認刪除」後，確認按鈕立即進入停用狀態（disabled）；若刪除請求因網路問題或後端錯誤失敗，系統顯示錯誤訊息並重新啟用按鈕，允許重試。
- 平行執行中若某個案例發生嚴重錯誤導致資源鎖死，系統應自動逾時終止該案例並繼續執行其餘案例，同時在報告中記錄異常。
- Robot Framework subprocess 執行途中因非逾時原因（如記憶體不足、程序崩潰、訊號終止）意外停止時，系統應捕捉異常並標記該案例為「執行失敗（系統錯誤）」，記錄崩潰原因，繼續執行清單中的其餘案例，確保部分失敗不阻斷整體執行流程。
- AI 代碼生成若因步驟描述過於模糊無法翻譯，系統應標記該步驟為「無法生成」並允許測試人員手動補充，不阻擋其他步驟的執行。
- 立即試跑失敗或逾時時，系統應顯示錯誤原因，不影響已儲存的測試案例資料，測試人員可選擇修改步驟後重試或略過試跑。
- AI 將測試步驟翻譯為 Robot Framework 程式碼時，若呼叫逾時（超過 35 秒，對應 SC-010）或 LLM API 回傳錯誤，程式碼預覽區應顯示明確錯誤提示（如「翻譯失敗，請稍後重試」），不清除測試人員已填寫的自然語言步驟內容，測試人員可選擇重試或略過翻譯直接儲存案例。
- 後台「目前使用模型」下拉僅列出目前可用模型；若已選定的啟用模型其 provider 事後變為不可用（`.env` 金鑰移除／Ollama 離線），AI 補齊／RF 代碼生成應顯示明確錯誤，管理員可於後台重新選擇可用模型。DB 尚未選擇時採 `.env` 的 `DEFAULT_LLM_MODEL`，其 provider 未設定金鑰時亦於呼叫時報錯。
- 本地 Ollama 未啟動或 `OLLAMA_BASE_URL` 無法連線時，`/llm-models` 的 Ollama 區塊應為空（不阻斷雲端模型列出）；若目前啟用模型為某 `ollama:` 模型但 Ollama 不可用，AI 補齊／RF 代碼生成應顯示明確錯誤（如「Ollama 服務無法連線」），不影響已填寫內容。
- 上傳 Excel/CSV/文字檔格式不符或欄位對應無法解析時，系統應顯示具體錯誤說明（如「第 3 欄缺少必要標頭」），不覆蓋已填入的測試資料。
- 執行結果頁面中若有截圖或影片檔案遺失或損毀，系統應顯示占位提示，不阻擋其餘結果資料的載入與呈現。
- 測試案例歷史執行紀錄為空（從未被任何清單執行過）時，系統應顯示「尚無執行紀錄」而非錯誤畫面。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須提供測試案例建立與編輯畫面，支援以下欄位：編號、名稱、說明、測試資料（以**四欄表格**呈現：**易讀名稱**（自由文字）、**RF 變數**（預設自動帶入 `${易讀名稱}`，可手動覆寫為自訂 RF 識別名稱）、**預設值**、**說明（備註）**；支援行內**新增列**、**行內直接編輯**、**刪除列**，可動態多筆操作）、前置步驟、主要步驟、建立人員、修改人員、系統別、標籤、版本號。畫面採**兩個 Tab 分頁**結構：**Tab 1「基本資訊」**放置編號、名稱、說明、測試資料、前置步驟、系統別、標籤、建立人員等欄位，以及「儲存案例」按鈕；**Tab 2「測試步驟」**採左右分割版面——**左側**為 AI 對話區（Chat 介面），使用者透過底部輸入框與 AI 進行多輪對話以描述、補齊或修改測試步驟，AI 回應以對話氣泡方式累積顯示，對話歷史於儲存時一併保存，下次開啟可繼續對話；**右側**為 Robot Framework 程式碼預覽區，AI 每次回應後自動更新右側的 RF 程式碼，無需額外點擊；左右兩個面板**各自具備獨立垂直捲軸**（`overflow-y: auto`），面板高度填滿可視區域（`calc(100vh - 220px)`），左右可獨立捲動，互不影響；Tab 2 亦提供「儲存案例」按鈕；兩個 Tab 均提供儲存按鈕，使用者可在任意 Tab 儲存。此 Tab 分頁版面同時適用於「建立測試案例」畫面（/cases/new）與「編輯測試案例」畫面（/cases/:id 的編輯模式）；進入**編輯模式**時，若案例已有儲存的 RF 程式碼，系統自動解析程式碼中所有非 RF 內建的 `${VARIABLE_NAME}` 參照（排除 `${CURDIR}`、`${SUITE_NAME}`、`${TEST_NAME}`、`${SUITE_DOCUMENTATION}` 等 RF 框架內建變數），採**合併補齊**策略自動新增尚未存在的測試資料列（`rf_variable` = `${VARIABLE_NAME}`，`field_name` = 去除 `${}` 後的識別名稱，`field_value` 與 `description` 預設空白），已存在的列不覆蓋；無已儲存 RF 程式碼時略過此步驟。
- **FR-002**: 系統必須在每次測試案例儲存時自動管理版本號（首次建立設為 v1，後續修改自動遞增）。
- **FR-003**: 系統必須提供測試案例清單頁面，以條列方式顯示所有案例的摘要資訊；清單頁面必須支援依 `case_number`、`name`、`system_category`、`created_at`、`version` 五欄點擊標題進行後端排序（Server-side），API 提供 `sort_by` 與 `order`（asc/desc）參數，資料庫層排序後回傳，確保跨頁排序結果一致；預設排序為 `created_at` 降序（最新建立優先）；表格標題顯示排序方向指示符（↑/↓），點擊同一欄標題切換 asc/desc。清單「操作」欄中的「編輯」與「刪除」按鈕必須採用與頁面外層一致的設計風格：「編輯」使用次要灰色樣式（灰底黑字）、「刪除」使用危險紅色樣式（紅底白字），兩者均採 border-radius 4px、padding 4px 10px、無外框，與全頁外層按鈕設計 token 一致。
- **FR-004**: 系統必須支援多條件篩選（至少包含系統別與名稱關鍵字），篩選條件可同時套用。
- **FR-005**: 測試人員必須能從清單點擊進入測試案例詳細畫面，查看完整欄位與版本資訊；詳細畫面採與建立/編輯畫面相同的 Tab 分頁結構：**Tab 1「基本資訊」**顯示案例完整欄位（唯讀），其中**測試資料以唯讀四欄表格**呈現（易讀名稱、RF 變數、預設值、說明）；媒體附件中**圖片與影片類附件**以可點擊下載連結呈現（指向後端 `GET /api/v1/media/attachments/{case_id}/{filename}`），點擊可直接於瀏覽器下載或預覽；**Tab 2「測試步驟」**左側顯示 AI 對話歷史（唯讀），右側顯示最新 AI 生成的 Robot Framework 程式碼（唯讀）；左右兩個面板**各自具備獨立垂直捲軸**（`overflow-y: auto`），面板高度填滿可視區域（`calc(100vh - 220px)`），各自可獨立捲動；瀏覽模式不提供編輯功能，需進入編輯模式（/cases/:id/edit）才可修改。
- **FR-006**: 系統必須允許測試人員建立測試清單，填寫清單名稱與建立人員，並從測試案例清單中選取案例匯入。**測試清單的整體頁面呈現比照測試案例管理**：（1）測試清單列表頁（/checklists）採用與測試案例清單頁（/cases）相同的版面結構（條列式、支援搜尋／篩選、可點入詳情）；清單列表頁必須支援依 `name`、`created_by`、`created_at`、`status` 四欄點擊標題進行後端排序（Server-side），API 提供 `sort_by` 與 `order`（asc/desc）參數，資料庫層排序後回傳；其中 `status` 為衍生欄位，依各清單最新一筆執行（排除已刪除）的狀態，採**自訂優先序**排序（asc 由上而下）：`completed` → `pending` → `running` → `failed` → `error` → 未執行（最後）；預設排序為 `created_at` 降序（最新建立優先）；表格標題顯示排序方向指示符（↑/↓），點擊同一欄標題切換 asc/desc；（2）測試清單詳情頁（/checklists/:id）採用與測試案例詳情頁相同的版面結構（標題區、基本資訊區、項目列表區、歷史紀錄區）；清單頁面必須顯示每次執行的歷史紀錄，包含執行時間、結果摘要，並可點入查看詳情；清單詳情頁（/checklists/:id）的案例列表中，每筆案例列提供**展開/收合切換鈕**（預設收合），點擊後向下展開顯示該案例的**測資變數表格**（4 欄：**易讀名稱**（唯讀）、**RF 變數**（唯讀）、**預設值**（唯讀，來自原案例）、**實際值**（可編輯，清單層級覆寫，持久化於清單-案例關聯））；允許多列同時展開，互不干擾；實際值於測試執行時優先套用（有填入時取代預設值），並持久化保留供下次執行使用。
- **FR-024**: 系統必須提供登入畫面（/login），使用者以帳號（username）與密碼登入；後端驗證成功後回傳 JWT Token，前端以 Bearer header 方式傳遞至後端 API；Token 未帶或無效時後端回傳 401，前端自動導向 /login；登出時前端清除 Token。所有需要登入才能存取的頁面均須在前端路由層進行 Token 檢查，未登入者導向 /login。
- **FR-025**: 系統必須提供管理後台頁面（/admin），僅限 `admin` 角色存取；後台帳號管理支援完整 CRUD：新增帳號（設初始密碼）、修改帳號資訊（含重設密碼）、指派角色（`admin`／`editor`／`viewer`）、停用／啟用帳號。三種角色的平台功能權限如下：（1）`admin` 擁有完整平台功能與後台管理存取；（2）`editor` 可建立、編輯、刪除測試案例與測試清單，以及執行測試清單；（3）`viewer` 可瀏覽所有案例與清單詳情，並可執行測試清單，但不可建立或編輯任何資料；未授權操作應顯示 403 提示。
- **FR-026**: 管理後台必須提供「系統別」管理功能，支援完整 CRUD：新增系統別、重新命名、刪除；刪除前顯示確認對話框，若有測試案例仍使用該系統別，系統顯示影響案例數供管理員確認後才執行刪除（軟刪除），已刪除的系統別不再出現於建立/編輯案例的系統別下拉選單。
- **FR-027**: LLM **金鑰／連線設定**由 `.env` 配置（唯讀、機密不落地）；**目前啟用模型**由管理後台選擇（持久化於 DB、即時生效）。
  - **金鑰／連線（`.env`，唯讀）**：`ANTHROPIC_API_KEY`／`OPENAI_API_KEY`／`OLLAMA_BASE_URL` 寫於 `.env`，後端啟動載入、變更需**重啟服務**。管理後台（/admin）LLM 分頁僅以**唯讀**呈現：金鑰顯示是否已設定及遮罩值（**前 7 字元 + `****…` + 末 4 碼**，如 `sk-ant-****…dF3a`；長度 < 12 整串 `****`），Ollama 顯示連線狀態與 `OLLAMA_BASE_URL`（非機密、可全文顯示）；**嚴禁**透過 API 回傳或顯示完整明文金鑰，後台不可編輯金鑰／連線。
  - **目前啟用模型（DB，可編輯）**：管理後台提供單一「目前使用模型」下拉，跨 provider（Claude／OpenAI／Ollama）**僅列出目前可用的模型**（雲端對應金鑰已設定、Ollama `/api/tags` 可連線）；選定後持久化於資料庫（`app_setting`）、**即時生效無需重啟**，全平台 AI 補齊與 RF 代碼生成（FR-012）採用此模型；DB 尚未選擇時 fallback 至 `.env` 的 `DEFAULT_LLM_MODEL`。此選擇為非機密指標（僅模型 id），不涉及金鑰落地。
- **FR-028**: 當 AI 生成 Robot Framework 測試腳本時，系統必須在生成 prompt 中注入 **CAPTCHA 自動識別規則**，確保面對含圖形驗證碼（CAPTCHA）的頁面時，生成的 .robot 檔能正確呼叫 `CaptchaSolverLibrary`。具體規則如下：（1）Settings 區塊需額外匯入 `Library    ${CURDIR}/../libs/CaptchaSolverLibrary.py`；（2）Keywords 區塊需定義「處理圖形驗證碼（如存在）」Keyword，採「若存在才處理」設計，截圖後以 AI Vision 辨識後填入輸入欄；（3）在填完帳號密碼後、送出前呼叫該 Keyword。`CaptchaSolverLibrary`（`libs/CaptchaSolverLibrary.py`）啟動時自動查詢後端 `GET /api/v1/llm-models` 取得目前啟用模型，依模型 ID 前綴路由至對應 provider：`claude-*` → Anthropic、`gpt-*`／`o1*` → OpenAI、`ollama:` → Ollama；後端不可連線時 fallback 至 `claude-haiku-4-5-20251001`；API 金鑰一律由後端 `.env` 讀取，不落地至 DB（遵循 FR-027 機密原則）。
- **FR-023**: 測試清單必須支援完整 CRUD 管理：（1）**編輯基本資訊**：可修改清單名稱與建立人員；（2）**刪除清單**：刪除前顯示確認對話框，若清單有執行中的測試則禁止刪除並顯示提示；（3）**案例管理畫面**（/checklists/:id/cases）：由清單詳情頁點擊「管理案例」按鈕進入，支援新增案例（從現有案例清單選取）、移除案例、調整執行順序（拖曳或上下移動按鈕）、為各案例新增清單層級備註；案例本身欄位在此畫面為唯讀，提供「前往編輯案例」連結跳轉至 /cases/:id/edit；**案例搜尋必須同時比對 `case_number`（案例編號）與 `name`（案例名稱）**，任一欄位包含關鍵字即顯示於搜尋結果；清單-案例關聯除備註外，亦儲存每個測資變數的**實際測試值**（`actual_value`），執行時優先套用（有值時取代案例的 `預設值`），無值時回退至案例預設值。
- **FR-007**: 系統必須允許設定外部資料庫連線，並提供連線測試功能；資料庫連線字串及認證憑證必須以加密方式儲存（後端使用金鑰加密後寫入資料庫），讀取時於後端記憶體內解密，嚴禁明文存放。**資料庫連線管理頁面（/db-connections）採用與測試清單列表頁相同的表格版面**：以可點擊的條列式表格呈現所有已設定的連線（欄位含連線名稱、資料庫類型、主機位址、狀態），支援搜尋欄位快速篩選，可點入各列查看連線詳情或進行編輯；整體結構比照 ChecklistsPage 設計（搜尋欄 + 表格 + 可點擊列）。
- **FR-008**: 系統必須允許從外部資料庫查詢撈取測試資料，並套用至測試案例的測試資料欄位。
- **FR-009**: 系統必須支援根據測試清單自動化執行網頁測試，並透過 **SSE（Server-Sent Events）** 顯示即時執行進度（前端以 `EventSource` 訂閱 `GET /api/v1/executions/:id/stream`，推送延遲上限為 2 秒，每 2 秒至少更新一次執行狀態）；進度事件來源為自訂 **RF Listener Plugin**（`--listener ExecutionListener`），監聽 pabot 執行中的 `start_test` / `end_test` 回呼，寫入共享 asyncio Queue 後由 SSE endpoint 讀取推送；允許不同使用者同時執行同一份測試清單，每次觸發各自產生一筆獨立的執行紀錄，互不干擾。
- **FR-010**: 系統必須在測試執行完成後產生包含每個案例結果的測試報告（通過/失敗、執行時間、失敗詳情）；RF 引擎原生產生的 `log.html`（詳細步驟記錄）與 `report.html`（執行摘要）必須於執行完成後以執行 ID 為鍵持久化儲存（不隨 temp 目錄刪除），由後端 API 提供查閱服務供前端 `<iframe>` 嵌入。
- **FR-011**: 測試報告必須支援匯出下載。
- **FR-012**: 系統必須整合多種 LLM 模型，初期必須支援 **Claude（Anthropic）**、**GPT-4o（OpenAI）** 與**本地 Ollama**，供測試人員在建立測試案例時呼叫 AI 理解並補齊測試步驟；**實際使用的模型為管理後台選定的「目前啟用模型」（持久化於 DB、即時生效，見 FR-027），未選時 fallback 至 `.env` 的 `DEFAULT_LLM_MODEL`；建立案例表單採用此全域模型，不提供個別模型選擇器**。
  - **Ollama（本地）**：以 `.env` 的 `OLLAMA_BASE_URL`（預設 `http://localhost:11434`）配置，有值即啟用、無需金鑰；採 Ollama **原生 API**（`POST /api/chat`，`stream:false`）呼叫，可用模型由 `GET /api/tags` **動態**取得；Ollama 模型於系統中以前綴 `ollama:` 標識（如 `ollama:gemma4:e4b`），後端依此前綴路由至 `OllamaProvider`。`/llm-models` 在 `OLLAMA_BASE_URL` 可連線時列出已安裝模型，查詢失敗則 Ollama 區塊為空、不影響雲端模型。
- **FR-013**: 系統必須允許測試人員在建立或編輯測試案例時上傳圖片、影片，或貼上網址，作為 AI 輔助步驟補齊的多媒體輔助資訊；上傳內容須與對應測試案例一併儲存。已上傳的圖片與影片附件必須可透過後端媒體端點（`GET /api/v1/media/attachments/{case_id}/{filename}`）下載，並在案例詳細畫面的「媒體附件」列表中以可點擊下載連結呈現（`<a href>` 加上 `download` 屬性）；URL 類型附件則以普通超連結呈現。
- **FR-014**: 系統必須允許測試人員刪除測試案例，依案例當前狀態採取以下三種行為：（1）**未被任何清單引用**：直接執行軟刪除；（2）**已被一個或多個清單引用但無執行中**：顯示受影響清單名稱並要求二次確認，確認後才執行刪除；（3）**有執行中的清單正在使用該案例**：禁止刪除，顯示「該案例正被執行中的清單使用，請等待執行完成後再刪除」提示，不顯示確認按鈕。所有確認刪除操作中，確認按鈕點擊後立即停用（disabled），直至刪除完成並關閉對話框，防止重複提交。
- **FR-015**: 系統必須支援測試清單的平行執行模式，允許同時執行多個測試案例，並在執行進度畫面中分別顯示各案例的即時狀態；後端採 **pabot（robotframework-pabot）** 執行平行測試，一次清單執行對應一個 `pabot --processes N suite1.robot suite2.robot ...` 指令，並行數由 `PARALLEL_MAX_WORKERS` 配置值（預設 5）控制；各案例的 .robot 檔由後端依清單內容動態產生，pabot 執行後以 output.xml 為來源解析結果並回寫至 Execution Record。
- **FR-016**: 系統必須透過 AI 自動將測試案例中的自然語言步驟翻譯為可執行的自動化測試代碼（輸出格式為 Robot Framework 語法），再由後端 Robot Framework + Playwright 引擎執行；翻譯結果與執行代碼版本須記錄於測試報告中。**RF 代碼生成時機**：使用者在 Tab 2 Chat 互動生成後，點擊「儲存 RF 程式碼」按鈕，系統將 RF 代碼以實體檔案形式儲存至 `robot_scripts/{case_number}.robot`（見 research.md Decision 8；採檔案而非 DB column，因 RF CLI 需要實體 .robot 檔）；執行時直接讀取已存在的 .robot 檔（零延遲）；若案例尚無已儲存的 .robot 檔（從未透過 Tab 2 Chat 生成並儲存），則於執行時觸發即時生成，受 SC-010 之 35 秒上限約束。
- **FR-017**: 系統必須在測試案例儲存後提供「立即試跑」功能，使用 AI 轉譯步驟並執行一次，並於畫面顯示結果（通過/失敗、截圖、執行時間）；試跑紀錄須與正式測試清單執行歷史明確區隔。若同一案例的試跑已在執行中，系統應拒絕新的試跑請求並顯示「該案例試跑正在執行中，請等待完成後再試」提示，不啟動新的試跑；試跑紀錄預設保留 30 天，逾期由系統自動清除（V2：管理員可於系統設定中調整保留天數；初期以 30 天固定值實作）。**Tab 2 立即試跑**：Tab 2「測試步驟」右側 RF 程式碼預覽區提供獨立的「立即試跑」按鈕；觸發時**直接使用右側預覽區當前顯示的 RF 程式碼**（不需先儲存案例）；執行完成後，系統將試跑結果以 `type: "trial_run_result"` 訊息附加至左側 Chat 對話區，訊息包含：通過/失敗 badge、錯誤訊息文字、執行時間、最多 1-3 張失敗截圖；系統同時自動觸發 AI 分析失敗原因並回應建議修正的 RF 程式碼；試跑結果訊息持久化至案例對話歷史（DB），與一般 Chat 訊息合併儲存，下次開啟案例可見；若右側 RF 程式碼預覽區無任何內容，系統應禁止試跑並顯示「RF 程式碼為空，請先透過 Chat 生成程式碼」提示。
- **FR-018**: 系統必須允許測試人員在測試案例的測試資料欄位，透過上傳 Excel、CSV 或文字檔案的方式批次匯入測試參數；上傳後提供預覽確認畫面，當測試案例已有既有測試資料時，預覽畫面須提供「追加（保留既有並新增）」與「取代（清除既有並改用匯入資料）」兩個選項供測試人員選擇，確認後才寫入欄位；若上傳檔案中存在無法對應已知測試資料欄位的標頭，系統顯示具體錯誤說明（如「第 N 欄標頭『XXX』無法對應」），允許使用者確認跳過未知欄位後繼續匯入，或取消匯入，不覆蓋既有資料；完全無法識別標頭的檔案則拒絕匯入。
- **FR-019**: 系統必須提供測試結果的專屬網頁呈現畫面，以測試引擎執行後產生的結構化報告為來源，顯示每個案例的執行結果（通過/失敗、執行時間、失敗詳情）；執行結果頁面必須新增「RF 報告」Tab，以 `<iframe>` 方式嵌入 RF 原生報告，Tab 內提供「詳細記錄 (log.html)」與「執行摘要 (report.html)」兩個子分頁切換；此「RF 報告」Tab 同時適用於清單執行結果頁（/results/:execution_id）與試跑結果頁，行為一致。
- **FR-020**: 測試結果頁面必須能內嵌顯示執行過程中擷取的截圖與影片，按測試步驟順序排列，無需另行下載即可在頁面上直接瀏覽。
- **FR-021**: 系統必須允許測試人員從測試案例清單或詳細畫面，快速導覽至該案例的所有執行歷史紀錄，並可點入查看各次執行結果詳情。
- **FR-022**: 系統必須在所有頁面提供全域導覽列，包含「測試案例管理」（/cases）、「測試清單」（/checklists）及「資料庫連線」（/db-connections）的快速連結，使測試人員可在任意頁面一鍵切換主要功能區塊，並以視覺方式標示目前所在頁面；在寬度小於 768px 的行動裝置或小螢幕環境下，導覽列自動收合為漢堡選單（☰），點擊後展開顯示完整導覽項目；全域介面字體採用 Noto Sans Traditional Chinese（透過 Google Fonts 載入）。

### Key Entities

- **使用者 (User)**: 平台登入帳號，包含使用者名稱（唯一）、密碼雜湊（bcrypt）、角色（`admin`／`editor`／`viewer`）、啟用狀態（`is_active`）、建立時間；由管理員透過後台建立與管理；刪除或停用帳號時不影響其建立的測試案例與清單資料。
- **測試案例 (Test Case)**: 可重複使用的測試單位，包含編號、名稱、說明、測試資料（多筆）、前置步驟、主要步驟、系統別、標籤、版本號、建立/修改人員，以及零至多份多媒體附件（圖片、影片、網址）作為 AI 輔助上下文，以及一份 AI 對話歷史。編號由「系統別+三位數序號」組成，序號自 001 開始往後遞增，在同一系統別內保持唯一性（例：Web-001、Web-002；API-001、API-002）。
- **AI 對話歷史 (AI Chat History)**: 附屬於測試案例的多輪對話紀錄，包含使用者訊息與 AI 回應，依時間順序排列；每條訊息包含角色（user/assistant）、內容文字、時間戳、**訊息類型（`type`：`chat` 一般對話 ｜ `trial_run_result` 試跑結果）**；`trial_run_result` 類型訊息含試跑結果摘要（狀態、錯誤訊息、執行時間）與最多 1-3 張失敗截圖的參照，AI 後續回應可存取此訊息以提供修正建議；與測試案例一併儲存，支援跨 session 延續對話。
- **測試清單 (Test Checklist)**: 由多個測試案例組成的執行集合，包含清單名稱、建立人員，以及多筆執行歷史紀錄；清單-案例關聯（ChecklistCase）除排序與備註外，亦儲存每個測資變數的**實際測試值**（`actual_value`，以 `variable_name → value` 映射持久化）。
- **測試資料 (Test Data)**: 附屬於測試案例的輸入資料集，可手動輸入或從外部資料庫匯入。
- **資料庫連線 (DB Connection)**: 外部資料庫的連線設定，用於自動化撈取測試資料。
- **執行紀錄 (Execution Record)**: 測試清單每次執行所產生的紀錄，包含執行時間、執行人員、結果摘要，關聯至詳細測試報告。
- **測試報告 (Test Report)**: 記錄單次執行結果的詳細文件，包含案例通過率、執行時間與失敗詳情。
- **媒體附件 (Media Attachment)**: 附屬於測試案例的多媒體資源，可為圖片、影片或外部網址，用於提供 AI 輔助步驟補齊時的視覺與語意上下文。
- **自動化測試代碼 (Automation Code)**: 由 AI 根據測試案例的自然語言步驟自動生成的可執行代碼，與測試案例版本關聯，記錄於每次執行報告中。
- **執行結果媒體 (Execution Media)**: 測試執行過程中自動擷取的截圖或影片，與對應的測試步驟關聯，顯示於測試結果頁面。

## Success Criteria *(mandatory)*

### Measurable Outcomes

> **量測基準**：所有時間類 SC 均以**使用者感知時間**（從操作觸發至 UI 顯示回應完成）為基準；除非另有說明，量測環境為後端 2 核心 / 4GB RAM、Chrome 最新版、網路延遲 < 50ms（本地或同資料中心）。

- **SC-001**: 測試人員可在 5 分鐘內完成建立一個包含所有欄位的測試案例。（量測方式：5 名測試人員首次操作計時，從進入建立畫面至點擊儲存完成，取中位數）
- **SC-002**: 多條件篩選後，測試案例清單在 2 秒內顯示結果（千筆案例規模）。
- **SC-003**: 測試案例版本號在每次修改儲存後準確自動更新，正確率達 100%。
- **SC-004**: 90% 的測試人員無需額外協助即可成功完成第一次測試清單建立與執行。（量測方式：招募 10 名目標使用者進行首次使用測試，記錄各自能否獨立完成完整流程而不需外部協助，以成功人數比例計算）
- **SC-005**: 測試報告在所有案例執行完成後 30 秒內可供查閱（報告頁面完整載入）或觸發下載（點擊匯出後瀏覽器開始接收資料），兩種操作均以 30 秒為上限。
- **SC-006**: 平台支援至少 10 位測試人員同時操作，p95 頁面操作回應時間仍維持在 2 秒以內（千筆案例規模；與單使用者情況相比無超過 20% 效能衰退）。
- **SC-007**: 測試人員使用 AI 補齊功能後，從觸發到取得建議結果在 15 秒內完成（正常網路環境下）。**此門檻以雲端模型（Claude／OpenAI）為準**；本地 Ollama 因推論速度受部署硬體影響，不適用此時間門檻。
- **SC-008**: 90% 的測試人員無需額外說明即可成功完成一次 AI 輔助步驟補齊操作。
- **SC-009**: 啟用平行執行時，整體測試清單的執行時間較循序執行縮短至少 40%（以包含 10 個以上案例的清單為基準）。
- **SC-010**: AI 代碼生成從觸發到完成在 35 秒內結束，且生成失敗率低於 5%。**此時間門檻以雲端模型（Claude／OpenAI）為準**；本地 Ollama 因推論速度受部署硬體影響，不適用此時間門檻（失敗率要求仍適用）。
- **SC-011**: 立即試跑功能從觸發到顯示結果在 60 秒內完成（單一案例、正常網路與目標環境可存取情況下）。
- **SC-012**: 透過 Excel/CSV 匯入測試參數時，系統解析與預覽載入在 5 秒內完成（單次上傳不超過 1000 筆資料列）。
- **SC-013**: 測試結果網頁在執行完成後 10 秒內可正常載入並顯示（含前 50 個截圖縮圖，影片為點擊後串流）。
- **SC-014**: 測試人員可在 3 次點擊內從測試案例清單導覽至任一執行結果的詳細頁面。

## Assumptions

- 平台為 Web 應用程式，測試人員透過瀏覽器存取，無需安裝本地端軟體。
- 使用者身份驗證（登入/登出與角色權限管理）已納入本平台規格範圍（見 FR-024、FR-025）；平台以 JWT Token 驗證機制保護所有 API 端點，未驗證或 Token 無效的請求自動導向 /login；初始管理員帳號由部署時透過後端腳本建立（seed）。
- 「系統別」為可管理的分類標籤，由管理員透過後台（FR-026）進行 CRUD 管理；初始分類由部署時 seed 資料提供。
- 測試案例的「主要步驟」採用自然語言格式，由使用者自由填寫；系統透過 AI 解析自然語言步驟來驅動網頁測試自動執行。
- 外部資料庫串接以 SQLite 為優先支援目標，未來可視需求擴充至其他資料庫類型。
- LLM 模型整合採用 API 呼叫方式，平台不自行訓練模型；初期支援 Claude（Anthropic）、GPT-4o（OpenAI）雲端模型與**本地 Ollama**（自架，無金鑰）；雲端金鑰與 Ollama base URL 皆由 `.env` 配置（見 FR-027），不由個別測試人員管理。Ollama 服務需由部署者自行啟動並 `pull` 所需模型，不在本平台管理範圍。
- 多媒體附件（圖片、影片）的檔案大小與格式限制由系統設定決定，合理預設為圖片最大 10MB、影片最大 100MB；網址僅需有效性驗證，不做全文抓取。
- AI 補齊功能為輔助性質，測試人員對最終步驟內容有完整編輯與覆寫權限。
- 平行執行的最大並行數量由系統設定決定，合理預設為同時執行 5 個案例，管理員可調整上限。
- AI 代碼生成採用與 AI 步驟補齊相同的 LLM 整合機制，生成結果與對應版本的測試案例一同快取，避免重複生成相同版本。
- 測試案例的刪除採軟刪除原則（保留歷史紀錄），已刪除案例不再顯示於清單，但其關聯的執行報告歷史仍可查閱。
- 「立即試跑」為非強制步驟，測試人員可選擇跳過；試跑結果僅供確認用途，不影響測試案例的正式狀態或版本。
- Excel/CSV/文字檔匯入採欄位對應機制，系統依標頭名稱比對測試資料欄位，不強制要求固定欄序；支援的格式：Excel（.xlsx/.xls）、CSV（UTF-8）、純文字（Tab 分隔）。
- 測試執行中的截圖與影片擷取為測試引擎的預設行為，擷取頻率與儲存規則由系統設定決定；測試人員無需手動觸發擷取。
- 自動化測試由後端伺服器上的測試引擎（**pabot + Robot Framework + Playwright**）執行；平行執行採 pabot（robotframework-pabot）管理，單一清單執行以一個 pabot 指令啟動所有案例的 .robot 檔；後端伺服器須預先安裝 Playwright 瀏覽器二進位檔（Chromium），並能以 headless 模式運行。測試人員的本地機器無需安裝任何瀏覽器驅動程式。
- 平行執行時，後端伺服器同時啟動多個 headless 瀏覽器實例；部署環境須具備足夠的 CPU 與記憶體資源（建議至少 2 核心 / 4GB RAM，預設並行數 5 時）。
- 測試結果頁面以網頁形式呈現，影片採串流方式提供預覽，無需完整下載即可播放；截圖以縮圖形式列出，點擊後顯示原始尺寸。
- 從測試案例視角查看執行歷史，顯示的是「含有此案例的所有清單執行紀錄」，而非獨立的單案例執行。
